# 业务操作系统

## 四层输入

| 层 | 作用 | 不得做什么 |
|---|---|---|
| 项目证据 | 说明当前业务真实发生了什么 | 不因获得批准而升级证据 |
| 行业与平台资料 | 补充角色、规则、替代和语言 | 不写成课程普遍规律 |
| 有效课程原理 | 发现问题、跳步和常见误区 | 不直接推出项目结论 |
| 决策工程 | 构造候选、比较、反证和持续学习 | 不制造隐藏事实或固定阈值 |

## 最早瓶颈原则

不要按资料出现顺序或课程章节推进。判断哪个未知或错误判断最可能使后续工作整体失效，从该处开始：

- 产品和购买任务未成立，从定位开始；
- 目标人已成立但内容无法推动明确跃迁，从内容开始；
- 脚本有效但无法稳定拍摄，从拍摄开始；
- 素材很多但旅程缺口明显，从内容配比开始；
- 内容资产有效但触达、承接或学习闭环断裂，从发布运营开始。

## 决策比较

高影响决策至少比较两个可行候选和维持现状。逐项记录：目标结果、支持证据、反证、预计成本、时间、风险、可逆性、承接容量、直接下游影响、30天表现、90天二阶效应。没有明显胜者时设计最小区分实验，不靠措辞强行选优。

## 跨阶段依赖

使用统一关系：

`定位判断 → 目标人/购买阶段 → 内容跃迁假设 → 脚本 → 拍摄资产 → 内容组合 → 渠道/承接 → 观察 → 修正最早因果节点`

下游成果必须声明依赖的上游 claim/decision。反馈异常时先定位最早可能出错的节点：

- 错的人大量进入：优先检查定位、目标人或内容筛选；
- 对的人观看但不行动：检查关卡机制、证据和 CTA；
- 内容方向正确但无法持续产出：检查拍摄形式与产能；
- 单条有效但整体转化弱：检查旅程内容配比和承接；
- 指标上升但成交无改善：检查指标与购买结果是否错配。

不得因为一个末端指标不好就推倒全部上游，也不得只改标题掩盖上游问题。

## 失效后的即时重建

`invalidate` 只完成风险隔离，不等于业务交付完成。新证据足以区分修订方向时，立即以新 claim/decision 为上游，按依赖顺序重建受影响的最小完整链：当前阶段成品 → 直接下游语义 → 可执行验证包。无需重新询问已登记事实，也不重写无关成果。

重建后必须对应登记每一件新成果，再执行 `validate`。“文件已生成但 artifact registry 不可见”属于跨阶段断链，不得交付 HANDOFF 或宣称当前链已重建。

例如购买角色链变化后，不能只提交影响清单；至少要把角色链、完整脚本、CTA、拍摄语义、内容组合和运营观察口径修订到可内部测试。若付款理由仍未知，交付具体的脱敏节点采集与验证机制，不用抽象“继续研究”占位。

只有以下情况允许停在影响清单：新证据相互冲突；缺失信息会改变两条以上可行修订路径；下一步需要真实拍摄、发布、外联或其他未授权动作；安全、隐私、医疗或法律边界未解决。

## 30天与90天

每次高影响决定同时输出：

- 30天：要完成的动作、领先指标、检查点、容量和停止规则；
- 90天：预期心智积累、内容供给、渠道依赖、承接压力、潜在副作用和需要提前准备的能力。

90天内容是情景预测，不是承诺。至少给出基准情景、积极情景和风险情景，以及触发调整的可观察信号。

## 结果优先

先交付可直接使用的定位、矩阵、脚本、分镜、排期、配比或运营卡，再附证据与控制说明。若证据不足，交付验证版成品并明确它验证什么，不能只给用户一张待填空表。
