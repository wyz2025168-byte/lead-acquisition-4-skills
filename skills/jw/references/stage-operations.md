# 发布运营系统

调用：`conversion-journey`、`orchestration-control`，并依据最早异常节点回调其上游方法；研究、反证和安全协议始终适用。

## 目标

把有效内容资产交给正确渠道和承接流程，用真实反馈判断哪条业务假设成立，并将修正返回最早因果节点。运营不是机械排期，也不能替代生产批准。

## 发布前

1. 校验内容、人物、素材、渠道、对象、时间和 CTA 是否拥有范围匹配的生产批准。
2. 校验宿主是否有可用发布工具。无工具时只生成执行清单，明确 `NOT_EXECUTED`。
3. 为每个渠道说明目标人为何在此、使用场景、内容适配、发布频率、承接入口和禁止动作。
4. 设计30天最小运营周期：固定项、唯一变量、负责人、节奏、指标、检查点、停止和扩展条件。
5. 预判90天的内容供给、人物容量、渠道依赖、咨询承接和心智累积风险。

## 指标树

按因果层级分开：

- 触达：目标人是否有机会看到；
- 识别：是否说出自己的具体阶段；
- 理解/信任：是否复述机制或引用真实信号；
- 行动：是否完成明确 CTA；
- 承接：是否到达咨询、预约、到场或方案节点；
- 业务结果：正价成交、收入、毛利、周期、退费或客诉。

点赞、播放、泛评论和普通问价是辅助指标，不得自动解释成购买意愿。

## 反馈归因

出现异常时生成至少两个解释，使用最小新证据区分：

- 目标人错误或阶段错误 → 返回定位；
- 对的人留下但内容无跃迁 → 返回内容；
- 内容正确但资产表达失败 → 返回拍摄；
- 单条有效但旅程断层 → 返回内容配比；
- 触达、承接、响应或记录失败 → 留在运营。

运行 `invalidate` 只让相关后代失效。记录真实观察、反证、用户纠正、重复问题和计划/实际差异。

## 交付物

- 渠道、对象、版本、时间、CTA 和承接责任人；
- 30天发布日历与周检查点；
- 指标树、记录口径和误导指标；
- 实验卡与继续/停止/扩展阈值；
- 基准、积极、风险三种90天情景；
- 真实执行日志或明确的未执行说明；
- 最早问题节点和下一轮唯一修正。

## 外部动作边界

不因“可以试试”获得生产批准；不超出批准的渠道、名单、素材、版本和期限；不静默投流、群发、联系客户或改变发布内容。执行失败必须原样报告，不得用计划稿冒充已发布。
