# 业务原子节点与因果链

五个阶段只负责向用户解释项目位于哪里；B0–O1 才是运行时门禁。

| 节点 | 判断 | 批准前必须具备 | 下游 |
| --- | --- | --- | --- |
| B0 | 真实结果合同 | 正价/等价价值交换、领先和排除事件、归因窗 | B5、C1、O1 |
| B1 | 业务/产品边界 | 产品、价格口径、容量、利润与不可承诺项 | B2–B5 |
| B2 | 交易角色链 | 使用、付款、决策、影响和否决角色 | B3–B5 |
| B3 | 触发与阶段 | 触发、现阶段、下一不可逆动作和替代 | B4–B5 |
| B4 | 线上可转化 | 可触达、可识别、可影响、可归因 | B5 |
| B5 | W1/W2 | 目标人、目标阶段、排除对象、三个候选；B6/B7 预检 | B6、B7、C1 |
| B6 | W3/W4 | 选择与信任理由、客户侧证据、反证 | B7、C1 |
| B7 | W5/W6 | 行动障碍、付款理由候选、最小验证 | C1、O1 |
| C1 | 内容策略 | 内容角色、跃迁机制、证据和选题边界 | C2 |
| C2 | 四圈与脚本 | 同一目标人贯穿开头、画面、正文和观点 | P1 |
| P1 | 生产呈现 | 人物、场景、形式和资源与上游一致 | O1 |
| O1 | 组合与运营 | 领先/成交指标分账、反馈和停止规则 | 反馈回 B0–P1 |

状态只有 `UNKNOWN / HYPOTHESIS / NEEDS_EVIDENCE / NEEDS_APPROVAL / APPROVED / INVALIDATED / SUPERSEDED`。未知并不等于停止；应生成候选和最小验证。批准下游前必须运行 `preflight`。B5 的 W3–W6 预检不是提前批准 B6/B7，而是要求至少两个候选、反证和最小验证，避免从“有需求”直接锁人。

一个回合默认只推进一个最高影响节点。上游被推翻时，从节点依赖图传播到真正后代；来源和无关事实不失效。
