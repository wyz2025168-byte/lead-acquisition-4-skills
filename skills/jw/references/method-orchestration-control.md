# 项目编排、审批、反馈与失效传播｜完整判断程序

- 唯一 owner：`$jw`
- origin：全部为 `AI_ENGINEERING`
- 课程来源权限：S02/S03 只约束锁—圈—成—生主链次序；S12/S13 只做 validation-only 回放
- 核心问句：“当前项目应调用哪个 owner、继承哪个批准版本、哪些后代已失效、唯一下一步是什么？”
- 正常产物：持久化状态和 `orchestration_summary`

## 1. 概念定义与适用边界

项目编排与失效传播方法 是工程编排能力，不是新增课程。它维护 owner、来源、版本、哈希、依赖、审批、stale、闸门和下一步；只读取并传递 定位与竞争选择方法–拍摄呈现与批量生产方法 的成果，无权创造、改写或批准其业务判断。适用：项目初始化/恢复、跨能力路由、上游变更、审批登记、局部失效、交接。不适用：补写课程缺口、用案例数字建阈值、直接生成策略或内容。

## 2. 必需事实与证据等级

1. `.jw-project` 的唯一当前状态、实体、claims、decisions、approvals、experiments 与 artifact registry。
2. 每个成果的 owner、origin、source_refs、version、payload_hash、approval_type、depends_on、stale 状态。
3. 当前用户请求与已有事实是否重复、冲突或覆盖。
4. 当前闸门的 Definition of Done 与未完成条件。
5. 课程主链和影响传播图。

本方法不代替业务方法评价证据真假，但必须验证每条高影响判断已经携带证据等级、范围、时效、反证与置信度。

## 3. 从事实到结论的推导步骤

1. 首次运行执行 `python3 scripts/jw_project.py init PROJECT_PATH`；再次运行先 `validate` 与 `status`，禁止凭对话记忆重建状态。
2. 读取事实注册表，已有信息不得重复询问；冲突时追加新 claim/decision，不覆盖历史。
3. 将请求定位到最早方法：定位与竞争选择、正价人群与购买阶段、人群连续性、六问与客户关卡、长期理念与内容心智、拍摄呈现与批量生产；研究、反证、内容组装和行业协议只能横向支持。
4. 路由前验证所需上游版本、哈希、审批和 stale；任一不满足则停在最早缺口。
5. 所有状态变更只经 `jw_project.py`：`add-claim`、`add-entity`、`add-decision`、`add-approval`、`add-experiment`、`register-artifact`、`invalidate`；模型不得手写高风险控制字段。
6. 审批映射严格区分四类；只有明确“试试/测试/实验/试运行/继续试”才可推断 `EXPERIMENT_AUTHORIZED`，单独“继续推进”不得猜审批类型。
7. 上游改变时从变更节点沿依赖图标记受影响后代 stale，未受影响成果保持字节/版本不变。
8. 若发现人群、阶段、购买理由、理念或呈现漂移，定位最早偏差方法，禁止整库重写；运行失效传播后，若现有证据足以修订且请求仍要求推进，立即重建受影响的最小完整下游链，不能把影响清单当作最终业务成果。
9. 只有重建后的成果已登记，或存在会改变修订方向的真实阻断时，才运行 `handoff` 生成唯一当前状态、证据限制、批准内容、失效项和唯一下一步；不把报告通过冒充业务通过。

## 4. 关键因果关系

- 持久状态使新任务能恢复已确认事实，减少重复询问和对话漂移。
- 审批类型与证据等级分离，能防止“允许测试”被升级为“事实已证实”。
- 依赖图使上游否决只传播到后代，既避免继续使用旧结论，也避免无关重写。
- 结构校验只证明格式和引用完整；业务反证 FAIL 时最终仍必须 FAIL。

## 5. 容易混淆但不能替代的概念

- 来源/证据等级 ≠ 审批类型。
- `HYPOTHESIS_ACCEPTED` ≠ `FACT_CONFIRMED`。
- `EXPERIMENT_AUTHORIZED` ≠ `PRODUCTION_APPROVED`。
- 结构 PASS ≠ 业务能力 PASS。
- 当前对话记忆 ≠ 持久项目状态。
- 案例回放成功 ≠ 新项目结论正确。
- 路由到方法 ≠ 编排能力可以代替业务方法作答。

## 6. 正例、反例与错误惯性

正例：用户否决主人群后，系统记录新事实，运行 `invalidate` 使依赖旧人群的内容、拍摄、配比和运营成果 stale，保留定位和无关行业研究；若新主人群方向已由证据区分，则同轮生成修订脚本、CTA、拍摄语义、配比与运营验证卡。只有新方向仍无法区分时才停在最小补证任务。

反例：用户说“先批准吧，继续试试”，系统将正价人群假设改成已确认事实，并自动发布脚本。正确映射只允许实验，证据等级不变，也不产生生产批准。

错误惯性：模型手写版本/哈希；新任务重新问已注册性别/职务；只更新 HANDOFF 不更新事件账本；上游一变就全项目重写；用满分结构测试声称业务已验证。

## 7. 反事实问题

1. 如果当前聊天全部丢失，能否仅凭 `.jw-project` 恢复唯一状态？
2. 如果删掉一个独立人物，另一人物的路线和依赖是否仍完整？
3. 如果某 claim 被否证，哪些具体 artifact 必须 stale，哪些不应变化？
4. 如果用户只说“继续”，系统能否证明这是哪种审批？不能则不得猜。
5. 如果 Schema 全过但 critic FAIL，最终状态是什么？必须 FAIL/blocked at owner。

## 8. 信息不足时的研究/实验路径

编排器不自己研究业务。缺事实调用内部研究协议，缺因果或反证调用内部反证协议，策略可进入内部制作时调用内容组装流程，口腔上下文调用口腔行业研究协议。横向成果必须回到对应业务方法形成判断，并经正确审批后才能登记为下游依赖。

## 9. 停止条件与下游交接

立即停止：运行时校验失败；高影响成果缺 method_refs、证据、反证、依赖、影响、审批或状态；审批语言歧义；上游 stale 且无法在本轮重建；真实发布未获 `PRODUCTION_APPROVED`；请求要求越过安全边界。

交接必须由 `python3 scripts/jw_project.py handoff PROJECT_PATH` 生成，至少显示：唯一当前状态、已确认事实、已接受假设、已授权实验、生产批准、stale 项、风险与证据限制、当前方法、唯一下一步。若证据已足以重建受影响链，交接前必须先完成重建；编排方法不输出额外课程结论。
