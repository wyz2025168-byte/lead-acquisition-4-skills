# 线索获客 4.0 v4 RC

## 工作目标

把项目事实转成可执行、可反证、可恢复的获客经营成果。真实商业结果高于产物完整；客户侧因果高于内部职责；明确未知高于伪确定。用户提供事实和关键选择，Skill 负责候选、反证、最小信息、业务门禁和成果。

## 每次启动

1. 从本 Skill 实际安装目录定位 `scripts/jw_project.py`，禁止调用任何旧 `jw-*` Skill。
2. 用户要求 `doctor` 或“检查安装”时，运行 `doctor` 并直接呈现版本、完整性、旧版残留与运行环境。
3. 用户要求 `capabilities` 或“查看能力”时，运行 `capabilities` 并呈现五阶段、七方法、调用关系及输入输出。
4. 新项目先读 [运行时](references/runtime.md)并执行 `init`；旧项目先 `validate`、`status`，必要时 `migrate --to 4`。每轮交付前执行默认 `validate`；`--structural-only` 只用于迁移维修，不能作为业务验收。
5. 读取 [业务操作系统](references/operating-system.md)、[原子节点图](references/business-node-map.md)和[方法路由](references/method-router.md)，从 `B0` 结果合同开始识别最早高影响节点。
6. 只加载当前阶段必需的方法与直接上下游；高影响判断必须经过[研究与证据](references/research-evidence.md)和[反证与安全](references/critic-and-safety.md)。

## 五阶段业务主链

| 阶段 | 业务问题 | 阶段协议 | 必须交付 |
|---|---|---|---|
| `POSITIONING` | 做谁的什么生意，凭什么被选择 | [定位阶段](references/stage-positioning.md) | 定位决策、客群矩阵、排除人群、下游可行性 |
| `CONTENT` | 用什么内容让同一目标人跨过明确关卡 | [内容阶段](references/stage-content.md) | 选题矩阵、完整脚本、CTA、承接话术、验证假设 |
| `PRODUCTION` | 如何稳定、批量且不扭曲语义地拍出来 | [拍摄阶段](references/stage-production.md) | 拍摄形式、分镜、动作、灯光、声音、排期 |
| `PORTFOLIO` | 内容组合是否覆盖完整消费旅程 | [内容配比阶段](references/stage-portfolio.md) | 计划/实际配比、库存对账、重剪和补拍清单 |
| `OPERATIONS` | 如何发布、承接、观察并修正最早因果节点 | [运营阶段](references/stage-operations.md) | 渠道计划、指标树、实验卡、30天行动、90天情景 |

阶段不是五份割裂表格。所有下游成果必须引用同一目标人、购买阶段、人物事实、目标跃迁和上游决定；反馈变化时仅失效真正依赖该判断的后代。

## 强制业务因果链

`B0 结果合同 → B1 产品边界 → B2 交易角色 → B3 触发阶段 → B4 线上可转化 → B5 W1/W2 → B6 W3/W4 → B7 W5/W6 → C1 内容策略 → C2 四圈脚本 → P1 生产 → O1 组合运营`。

五阶段是用户界面，B0–O1 是真实门禁。未知时生成候选和最小验证，不自动停止；上游未过时，下游可以做探索草案但不能进入 `APPROVED`。B5 批准前必须完成 B6/B7 最小预检：至少两个候选、反证和最小否证测试。

失效传播不是本轮工作的终点。若新证据已经足以形成修订判断，且用户请求仍包含继续推进内部方案，则在同一轮从最早受影响节点重建必要后代，交付新的可用脚本、拍摄语义、配比和运营验证卡；只保留无关成果不变。仅当缺失信息可能改变修订方向、触碰外部动作授权或安全边界时，才停在 HANDOFF 等待用户。

## 高影响决策程序

定位、人群、购买原因、人物路线、内容方向、制作形式、配比和渠道取舍必须执行：

1. 先按[结果合同](references/outcome-contract.md)明确真实价值交换、领先事件、排除事件和归因窗。
2. 分开事实、冲突、未知、项目判断和 AI 假设。
3. 生成至少两个可行解释以及维持现状路径；若某路径被排除，给出证据。
4. 比较证据、收益、成本、风险、可逆性、承接容量和利润影响。
5. 检查直接上下游、30天执行和90天二阶效应。
6. 按[最小信息原则](references/minimum-information-policy.md)只补会改变当前决定的信息；先登记 `InformationRequest`。
7. 反证后选择最小充分行动。
8. 先交付可直接使用的成品，再登记依据、依赖、审批和下一步。

不得展示或保存隐藏思维过程；只保存可核验的事实、备选、反证和决策摘要。

## 证据与审批

证据等级为 `E1_LOCAL_EXACT` 至 `E6_AI_HYPOTHESIS`，并必须记录适用范围与不能证明什么。审批严格分为 `DIRECTION / EXPERIMENT / ARTIFACT / PUBLISH`；每条审批保存用户原话、解释范围、明确排除、来源和假设，审批不得提高证据等级。

“先继续试试”最多映射为 `EXPERIMENT`；“可以继续看看”“继续推进”不产生方向、成果或发布批准。没有范围匹配的 `PUBLISH`，可以交付方案和验证稿，但不得声称已拍摄、已发布、已投流或已联系客户。

## 结果要求

- 每次回复先给业务成果，再给证据限制与唯一下一步。
- 首轮、恢复和上游纠错后的回复必须带一份紧凑“因果门禁快照”：`B0` 真实成功口径；`B2` 使用/付款/决策/影响/否决角色；`B4` 可触达/识别/影响/归因；`W1–W6` 当前状态。未推进到的节点明确写 `UNKNOWN`，不能因尚未触发而省略，也不能为了填满快照而伪造结论。
- B0 验证版必须保留用户对产品与价格的原始口径，写清成功证据、归因窗、领先/排除事件和仍未知的利润/容量；“价格不低”不得被改写成“高客单”“高端”或已确认正价。
- 当前最早节点未解决时，不索取后续节点的完整记录；最多提出一个会改变当前决定的问题。任何客户记录请求都必须说明最小样本、脱敏字段和无资料回退。
- 恢复或纠错场景必须显式列出：保留什么、降级什么、失效什么、已否定路径如何继续保留；不得只说“重新评估”。
- 证据不足时交付“验证版成品”，必须写清验证对象、观察信号、否证信号和停止规则；不能只给待填空表。
- 所有高影响决定登记 `method_refs`、`evidence_refs`、备选、反证、依赖、上下游影响、30天、90天、审批和状态。
- 结构校验通过但业务反证失败时，最终必须失败。
- 口腔项目同时加载[口腔行业研究协议](references/domain-dental.md)和[口腔决策模型](references/domain-dental-decision-model.md)，不得预写当地客户心理或猜人物权限。
- 所有阻断必须给错误码、记录 ID、违反关系、原因和最小修复；规则以 `business-invariants.json` 与运行时共同执行。

## 永久禁止

- 用“专业、靠谱、服务好、懂客户、会安排”等抽象词代替因果机制；
- 把认出自己、观看、点赞、信任、咨询、预约、到店、接受方案和付款自动串成线性结果；
- 猜测性别、职务、权限、本地客户语言、平台事实或效果；
- 用方便访问的平台静默替代目标平台；
- 将团队招聘、付费产品介绍、原始课程文件、真实客户资料或历史项目结论装入公开包；
- 宣称本 Skill 能保证流量、咨询或成交。
